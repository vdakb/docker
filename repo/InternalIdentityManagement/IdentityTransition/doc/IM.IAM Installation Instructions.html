2 Datensicherung

Bevor mit der Bereitstellung der Konnektoren begonnen wird, wird empfohlen eine vollständige Datensicherung der bestehenden Konfiguration
vorzunehmen. Diese Datensicherung erlaubt es den ursprünglichen Zustand wiederherzustellen, für den Fall, das während der Bereitstellung
irreversible Fehler auftreten.

3 Deployment Konfiguration

Für die halbautomatische Bereitstellung sind einige vorbereitende Schritte notwendig, um das Deployment durchführen zu können.
Diese Schritte sind im Einzelnen:

  1. Bezug der Ablaufsteuerung

  2. Festlegung der Konfiguration

  3. Bezug des Deployment

3.1 Bezug der Ablaufsteuerung

Die Steuerung des Deployment wird über das Version Control System bezogen.
Der Bezug wird durch folgende die Kommandozeilenfolge vorgenommen:

mkdir –p <working>
cd <working>
svn --username <username> export --force <sourceurl> .
Parameter Beschreibung
<working> Der Name des Verzeichnisses, in das der Bezug vorgenommen wird und aus dem heraus alle nachfolgenden Operationen gestartet werden.
<username> Der Benutzername der durch das Kommando svn für die Anmeldung am Versions Control System zu verwenden ist.
<sourceurl> Die URL zum Versions Control System über die die Ablaufsteuerung in des lokale Dateisystem exportiert wird.

Beispiel:
https://gcdpsource.oracle.com/svn/iam-bka/internal/1.0/trunk/IdentityTransition

Beispiel:

svn --username dieter.steding@oracle.com export --force https://gcdpsource.oracle.com/svn/iam-bka/internal/1.0/trunk/IdentityTransition .

Anschließend sollte das aktuelle Arbeitsverzeichnis gegen unberechtigte Zugriffe geschützt werden
Dazu wird folgende Kommandozeilen verwendet:

find ./ -type f -exec chmod 600 {} \;
find ./ -type d -exec chmod 700 {} \;

Anschließend ist für das zentrale Shell-Skript der Ausführungsmodus auf Benutzerebene freizugeben.
Dazu wird folgende Kommandozeile verwendet:

chmod u+x install installAll configAll bootstrapAll onboard onboard-identity onboard-account onboard-lookup

3.2 Festlegung der Konfiguration

Das Deployment kann von einem zentralen Server aus auf alle für diesen Server erreichbaren Systemkomponenten vorgenommen werden.
Dazu wurden aus dem Version Control System die Umgebungskonfiguration als Vorlagen exportiert.
Diese werden durch die nachfolgenden Schritte auf die jeweilige Zielkonfiguration angepasst.

  1. Anlegen und Einrichten des Wallet

  2. Anlegen und Einrichten der Parameter

  3. Substitution der Platzhalter in der Konfiguration

3.2.1 Anlegen und Einrichten des Wallet

Die Kennwörter werden über ein Wallet, welches durch Oracle Platform Security Services verwaltet wird, isoliert.
Das vorgegebene Verzeichnis für das Wallet ist:

./jps/identity

Zum Anlegen des Wallet wird das Werkzeug mkstore verwendet.
Der Aufruf zum Anlegen sieht folgendermaßen aus:

. iam.env
export PATH=${FMW_HOME}/oracle_common/bin:${PATH}
mkstore -wrl ./jps/identity/ -create

Es wird ein Passwort abgefragt, mit dem das Wallet vor unbefugtem Öffnen geschützt wird.

Die Einträge erfolgen wiederum mit mkstore nach dem Muster:

echo <credential> | mkstore -wrl ./jps/identity/ -create -wrl jps/identity -createUserCredential <stage> <alias> <password>
Parameter Beschreibung
<credential> Das Kennwort, das bei der Anlage des Wallet vergeben wurde.
<stage> Der Name der logischen Gruppe von Credential's.
<alias> Der Aliasname des Credential's welcher später im Schritt Substitution für die Extraktion des Kennworts verwendet wird.
<password> Das Credential welches im Schritt Substitution zu verwenden ist.

Beispiel:

echo Welcome1 | mkstore -wrl ./jps/identity -createUserCredential intdev scp@oig oracle Welcome!

Eine Aufstellung der notwendigen Aliase befindet sich im Anhang.

3.2.2 Anlegen und Einrichten der Parameter

Die Parameter für die Substitution der übrigen Parameter wird über Textdateien vorgenommen.
Das vorgegebene Verzeichnis für die Dateien ist:

./src/cfg

Zum Anlegen einer Datei wird die Vorlage intoig.properties verwendet.
Der Aufruf zum Anlegen sieht folgendermaßen aus:

mkdir -p ./src/cfg
cp ./tpl/cfg/intoig.properties ./src/cfg/<stage>.properties
Parameter Beschreibung
<stage> Der Name der logischen Gruppe von Parametern.

Die Namen dieser Dateien sollten der Namensgebung des Parameters <stage> beim Anlegen des Wallet folgen

Datein Beschreibung
intdev Deklaration der Umgebungsvariablen für die Entwicklungsumgebung des Internen Identity Managements.
intint Deklaration der Umgebungsvariablen für die Integrationsumgebung des Internen Identity Managements.
intedu Deklaration der Umgebungsvariablen für die Schulungsumgebung des Internen Identity Managements.
intppr Deklaration der Umgebungsvariablen für die Pre-Produktionsumgebung des Internen Identity Managements.
intppa Deklaration der Umgebungsvariablen für die Produktionsumgebung des Internen Identity Managements.

Beispiel:

mkdir -p ./src/cfg
cp ./tpl/cfg/intoig.properties ./src/cfg/intdev.properties

Anschließend sind die Werte für die Parameter in einem Texteditor auf ihre erforderlichen Werte anzupassen.

Eine Aufstellung der notwendigen Parameter befindet sich im Anhang.

3.2.3 Substitution der Platzhalter in der Konfiguration

Die Substitution der Platzhalter in den Konfigurationsdateien erfolgt über eine Shell Skript.
Für dessen Ausführung werden Umgebungsvariablen vorausgesetzt.
Diese Umgebungsvariablen werden vorzugsweise in einer Datei global für alle die Infrastruktur umfassenden Systemkomponenten erzeugt.
Somit wird sichergestellt, dass unabhängig von einem Systemkomponenten-Deployment alle notwendigen Informationen deklariert sind.
Vorzugsweise ist die Datei im Dateipfad des ausführenden Benutzers auffindbar.

Die empfohlene Lokation für diese Datei ist:

${HOME}/bin

Checkpoint

. <stage>
./install replace
Parameter Beschreibung
<stage> Der logische Name des Infrastrukturumgebung.

Beispiel:

. intdev

3.3 Bezug des Deployment

Ist das Deployment bereits vorhanden kann es mittels

./install verify

überprüft werden. Es wird eine Konsistenzprüfung der exportierten Konfigurationsdateien und eine Syntaxprüfung aller XML Dateien ausgeführt.

Alternativ wird das Deployment ebenfalls über das Version Control System bezogen. Der Bezug wird durch die Kommandozeile

./install update

ausgeführt.

Diese Operation führt neben dem Bezug des Deployment eine Konsistenzprüfung der exportierten Konfigurationsdateien und eine Syntaxprüfung aller XML Dateien aus.

4 Bereitstellungsprozess

Das Deployment ist ein Prozess, der in seinem Ablauf die Bereitstellung von Konfigurationen, Entitäten sowie den Bezug von Daten aus diversen
Quellsystemen umfasst. Dabei sind zu einem gegebenen Zeitpunkt Artefakte des Deployment auf bereist geladene Daten angewiesen, um
fehlerfrei im Oracle Identity Manager bereitgestellt werden zu können. Andererseits haben einzelne Artefakte Abhängigkeiten untereinander, die
die Einhaltung einer bestimmten Reihenfolge innerhalb des Bereitstellungsprozesses erfordern.

Der Bereitstellungprozess selbst untergliedert sich in wesentliche Teilprozesse:

  1. Setup

  2. Deployment

  3. Bootstrap

  4. Post Deployment

4.1 Setup

Im Teilprozess Setup werden die Server-Komponenten in ihrer Konfiguration geändert und miteinander verbunden.

Der Teilprozess Setup selbst untergliedert sich in wesentliche Schritte:

  1. Einbringen der Entwicklungartefakte

  2. Deployment der Entwicklungartefakte

4.1.1 Einbringen der Entwicklungartefakte

Der Schritt wird über die Kommandozeile

./install setup all

zur Ausführung gebracht.

Checkpoint

Nach diesem Prozess ist auf dem ausführenden Knoten durch pack eine neues WebLogic Domain Template für die WebLogic Domain iam zu erzeugen.
Auf jedem weiteren Knoten ist dieses Template durch unpack im Cluster bereitzustellen.

Siehe: Overview of the Pack and Unpack Commands

4.1.1 Deployment der Entwicklungartefakte

Für die Anpassung der Oberflächenelemente wird die Anwendungsbibliothek Identity Manager Customization Library bereitgestellt.

In diesem Schritt erfolgt die Einbindung dieser Anwendungsbibliothek in das Deployment des WebLogic Managed Servers (Cluster) in dem Oracle
Identity Manager bereitsgestellt wurde.

  1. In WebLogic Server Administration Console wird im Panel Domainstruktur über den Link Deployment in die Auflistung der existierenden
    Deployment der Domain gewechselt.

  2. Die Bereitstellung der Anwendungsbibliothek erfolgt über die Schaltfläche Installieren.
    Im darauffolgend angezeigten Assistenten wird in den entsprechenden Verzeichnispfad

    /opt/oracle/product/iam/12.2.1/idm/server/apps

    navigiert und die Datei bka.iam.ui.identity.library.war ausgewählt.

  3. Über die Schaltfläche Weiter wird in der Bereitstellung der Bibliothek fortgefahren.
    Als Art des Deployment ist sicherzustellen, dass die Option Dieses Deployment als Library installieren ausgewählt wurde.

  4. Über die Schaltfläche Weiter wird in der Bereitstellung der Bibliothek fortgefahren.
    Als Ziel des Deployment wird aus den verfügbaren Zielen der Cluster (bzw. Managed Server) ausgewählt, in dem Oracle Identity Manager
    bereitgestellt wurde. Innerhalb des Clusters wird die Option Alle Server im Cluster verwendet, um das Deployment auf alle Server
    vorzunehmen.

  5. Über die Schaltfläche Weiter wird in der Bereitstellung der Bibliothek fortgefahren.
    In den daraufhin angezeigten Optionalen Einstellungen sind ist in der Sektion Source accesibility die Option

    I will make the deployment accessibke from the following location

    ausgwählt wurde.

  6. Über die Schaltfläche Fertigstellen wird nun die Bereitstellung der Bibliothek vorgenommen.
    Die erfolgreiche Bereitstellung wird durch das System entsprechend bestätigt.

  7. Als Ergebnis dieses Vorgangs werden nun zwei Bibliotheken mit gleichem Namen und unterschiedlichen Versionsnummern in der Deployment
    Übersicht angezeigt. Allerdings wird zurzeit nur die Anwendungsbibliothek mit der niedrigeren Versionsnummer verwendet. Erst nach einem
    Neustart der Managed Server wird die Anwendungsbibliothek mit der höheren Versionsnummer entsprechend referenziert.

    Es muss nun noch die Deployment-Reihenfolge der Anwendungsbibliothek angepasst werden. Standardmäßig ist 100 für diese Reihenfolge
    festgelegt. Wichtig ist, dass bei Änderung dieses Wertes die Deployment-Reihenfolge so eingestellt wird, dass die Anwendungsbibliothek vor
    den eigentlichen Anwendungen bereit steht, die sie referenzieren und nach den Modulen, die die Anwendungsbibliothek selbst verwendet.

    oracle.iam.ui.custom(11.1.1,11.1.2)
    404

Achtung

Nach diesem Prozess ist der Managed Server der Systemkomponente

  • Oracle Identity Manager
  • Oracle SOA Suite
neu zu starten

4.2 Deployment

Im Teilprozess Deployment werden die Entwicklungsartefakte in den Oracle Identity Manager eingebracht und konfiguriert.

Der Prozess wird über die Kommandozeile

./installAll

zur Ausführung gebracht.

4.3 Boostrap

Nach der Bereitstellung der Basisentitäten werden die nach Anforderungsdefinition erforderlichen Identitäten und Berechtigungsobjekte in das
System eingebracht.

Der Prozess wird über die Kommandozeile

./boostrapAll

zur Ausführung gebracht.

4.4 Post Deployment

Nach der Bereitstellung der Basisentitäten sind manuelle Schritte notwendig, die das System in den Grundzustand versetzen, der für alle
weiteren Aktivitäten zwingend notwendig ist.
